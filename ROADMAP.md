# holodOS — Roadmap

Личный трекер продуктов в холодильнике. Идеи на будущее — в порядке примерной полезности и сложности.

---

## 1. Low-stock («мало осталось»)

**Суть:** продукт ещё в холодосе, но пора докупить. Может отображаться и в холодосе, и в списке покупок.

**Метод:**
- флаг `low_stock` в таблице `products`;
- отдельный жест «мало» (не путать с «+» = закончилось);
- список покупок: `finished` + активные с `low_stock`;
- визуальное различие «заканчивается» / «закончилось».

**Сложность:** низкая — одна колонка, правки хука и двух вкладок.

**Нужность:** высокая — закрывает реальный сценарий без перегруза системы.

**Этапы:** БД → хук → логика списка покупок → кнопка «мало» в холодосе → refresh вкладок → метки и подсказка.

---

## 2. Срок годности

**Суть:** отмечать срок годности, подсвечивать продукты, которые скоро испортятся.

**Метод (поэтапно):**
1. **MVP:** поле `expires_at`, усреднённый срок по категории при добавлении, ручная правка, подсветка «скоро» / «просрочено».
2. **Позже:** распознавание даты с упаковки по фото (OCR / vision API).

**Сложность:** средняя без OCR; высокая с фото (камера, API, ошибки распознавания).

**Нужность:** средняя–высокая для холодильника; для списка покупок — ниже. Усреднённые сроки — разумный старт; фото — nice-to-have.

**Зависимости:** логично после low-stock.

---

## 3. Рецепты

**Суть:** вкладка «Рецепты» — что можно приготовить из продуктов в холодосе. Только свои рецепты, вручную; без внешних библиотек, ссылок и импортов.

**Флоу пользователя:**
1. Открыть вкладку «Рецепты».
2. **Можно сейчас** — все ингредиенты есть в холодосе (`status = active`).
3. **Не хватает** — перечислены продукты, которых нет в холодосе (обычно уже в списке покупок, `finished`). Кнопка «купить» не нужна — список покупок уже закрывает этот сценарий.
4. **Нужно поправить** — рецепт «сломан»: один или несколько ингредиентов удалены из приложения.

**Карточка рецепта (создание и просмотр):**
- название;
- **тип приёма пищи:** завтрак / обед (`meal_type`, обязательное поле);
- ингредиенты — выбор только из **каталога продуктов app** (`product_id`, без свободного текста и дублей «яйцо / яйца»);
- текстовая инструкция;
- время приготовления (минуты).

**Завтраки и обеды по времени суток:**
- до **12:00** (локальное время устройства) на вкладке «Рецепты» **сверху** блок «Завтраки», ниже — «Обеды» (можно свернуть второй блок);
- с **12:00** — наоборот: сверху «Обеды», ниже «Завтраки»;
- оба типа **всегда доступны** (мягкий режим, не скрывать «неактуальный» тип полностью);
- внутри каждого блока — те же секции: «Можно сейчас» / «Не хватает» / «Нужно поправить».

**Метод:**
- таблицы `recipes`, `recipe_ingredients` (`recipe_id`, `product_id`);
- в `recipes`: `meal_type text check (meal_type in ('breakfast', 'lunch'))`;
- `product_id` → `ON DELETE SET NULL` (удаление продукта не удаляет рецепт, а помечает его как «нужно поправить»);
- готовность: все ингредиенты `active` и все `product_id` валидны;
- порядок блоков: `src/lib/recipeMealTime.ts` — `getPrimaryMealType()` по текущему часу, константа `BREAKFAST_UNTIL_HOUR = 12`;
- бинарная модель есть/нет — без граммовок, брендов и количеств.

**Улучшения (после MVP):**
- в «Не хватает» сортировка «почти готов» — сверху рецепты с наименьшим числом недостающих ингредиентов;
- подпись у недостающего: «в списке покупок» (информативно, без кнопки);
- после «купила» на списке покупок рецепт автоматически переезжает в «Можно сейчас»;
- CRUD рецептов: вкладка «Рецепты» + форма (как `ProductSettingsForm` — переиспользуемый компонент);
- позже: тип «ужин», настраиваемый порог времени в Settings.

**Сложность:** средняя — новая вкладка и домен, но без OCR, замен ингредиентов и внешних API.

**Нужность:** высокая для сценария «что приготовить из того, что есть»; хорошо стыкуется с бинарной моделью holodOS.

**Зависимости:** единый каталог продуктов; стабильный flow холодос ↔ список покупок.

**Этапы:** БД → хук `useRecipes` → вкладка и секции → форма рецепта → `meal_type` + группировка по времени суток → статус «нужно поправить» при удалении продукта → полировка (сортировка, подписи).

**Инструкция для реализации:** [RECIPES_IMPLEMENTATION.md](./RECIPES_IMPLEMENTATION.md)

**Не делаем в v1:** внешние рецепты, ссылки, замены ингредиентов, граммовки, кнопка «добавить недостающее в покупки», preset-библиотека, жёсткое скрытие «неактуального» типа приёма пищи.

## 4. Сезонные продукты

**Суть:** скрывать вне сезона (например, ягоды); возвращать в список по календарю.

**Метод:**
- конфиг `season_months` по продукту/категории;
- фильтр при добавлении и в каталоге;
- проверка по текущей дате при открытии приложения (без cron).

**Сложность:** низкая–средняя.

**Нужность:** низкая–средняя для личного холодильника — в основном убирает шум в длинном каталоге. Полезно при разросшемся списке продуктов.

---

## Рекомендуемый порядок (фичи)

| Приоритет | Фича | Почему |
|-----------|------|--------|
| 1 | Low-stock | мало кода, заметная польза |
| 2 | Срок годности (без OCR) | естественное продолжение учёта |
| 3 | Сезонность | если мешает длинный список при добавлении |
| 4 | OCR срока по фото | когда надоест ручной ввод |
| 5 | Рецепты | отдельный модуль; бинарная модель упрощает реализацию |

## UX: много продуктов в холодосе

**Проблема:** плоский список + фильтры по категориям — наглядно, но при 30+ позициях превращается в «простыню». Нужно облегчить UI, не теряя ориентацию по категориям и быстрый доступ к действиям (`+`, удаление).

**Принцип:** сначала поиск и визуальные якоря; сворачивание и «умное» скрытие — если первого мало.

---

### UX-1. Поиск по названию

**Суть:** поле ввода над списком; мгновенная фильтрация по имени продукта.

**Метод:** локальный `filter` по `products` в `ProductList`; фильтры категорий остаются и комбинируются с поиском.

**Сложность:** низкая.

**Нужность:** высокая при длинном списке — самый быстрый способ найти «молоко» без прокрутки.

---

### UX-2. Липкие заголовки категорий

**Суть:** тонкие разделители с названием категории, фиксируются при скролле (sticky). Сортировка по категориям сохраняется, без сворачиваемых блоков.

**Метод:** при рендере плоского списка вставлять заголовок при смене категории; CSS `position: sticky`.

**Сложность:** низкая–средняя.

**Нужность:** высокая — компромисс между плоским списком и старыми тяжёлыми секциями; всегда видно «где я» в списке.

---

### UX-3. Компактный режим карточки

**Суть:** опционально или по умолчанию — одна строка: `+` · название · категория мелким серым текстом. Меньше вертикального пространства.

**Метод:** вариант `ProductItem` (`compact`) или CSS-модификатор; переключатель в UI или настройках.

**Сложность:** низкая.

**Нужность:** средняя–высокая — в ~2 раза больше продуктов на экране; смысл карточки не теряется.

---

### UX-4. Сворачиваемые категории (только в холодосе)

**Суть:** вернуть группировку с заголовками и счётчиком, но умнее: по умолчанию свёрнуты «редкие» категории, развёрнуты частые или с low-stock.

**Метод:** состояние `collapsedCategories` + эвристика «развёрнуто по умолчанию» (по последнему действию или ручной настройке).

**Сложность:** средняя.

**Нужность:** средняя — если поиск + липкие заголовки не хватает. Список покупок можно оставить плоским.

---

### UX-5. Закреплённые / частые продукты

**Суть:** блок «Часто» вверху (5–10 позиций) — вручную или по частоте действий (`+`, «мало», добавление).

**Метод:** флаг `pinned` в БД или localStorage; отдельная секция над основным списком.

**Сложность:** средняя.

**Нужность:** средняя — полезно, если 80% внимания на одних и тех же продуктах; для разнообразного холодильника — слабее.

---

### UX-6. Секция «Редко используемые»

**Суть:** продукты без действий N дней — внизу списка или в свёрнутой секции «Редко», чтобы не мешали ежедневному обзору.

**Метод:** поле `last_touched_at` или вывод из `created_at` / истории; порог N дней в конфиге.

**Сложность:** средняя (нужно отслеживать активность).

**Нужность:** низкая–средняя — актуально при очень длинном списке и редких «забытых» позициях.

---

### UX-7. Виртуальный скролл

**Суть:** рендер только видимых строк при 100+ продуктах.

**Метод:** `react-window` или аналог (новая зависимость — только если действительно нужна).

**Сложность:** средняя; влияет на UX минимально (плавность, не структура).

**Нужность:** низкая на текущем масштабе — преждевременная оптимизация.

---

### Рекомендуемый порядок (UX холодоса)

| Приоритет | Идея | Почему |
|-----------|------|--------|
| 1 | Поиск по названию | быстрый эффект, мало кода |
| 2 | Липкие заголовки категорий | наглядность без тяжёлых блоков |
| 3 | Компактный режим карточки | больше продуктов на экране |
| 4 | Сворачиваемые категории | если 1–3 не хватает |
| 5 | Частые / закреплённые | персонализация под привычки |
| 6 | Секция «редко» | при очень длинном списке |
| 7 | Виртуальный скролл | только при сотнях позиций |

**Связь с фичами:** сезонность (п. 4 roadmap) полезнее для **формы добавления** и каталога, чем для основного списка в холодосе.
