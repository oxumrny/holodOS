# holodOS

Личный веб-трекер продуктов в холодильнике: что есть дома и что нужно купить. Проект для себя и близких; репозиторий: [github.com/oxumrny/holodOS](https://github.com/oxumrny/holodOS).

---

## Стек

| Слой | Технология |
|------|------------|
| Фронтенд | Vite 6 + React 19 + TypeScript |
| Бэкенд / БД | Supabase (PostgreSQL + REST API) |
| Стили | обычный CSS (файлы рядом с компонентами) |
| Деплой | Vercel (`vercel.json`, сборка в `dist/`) |

Раньше проект был **Expo + React Native**; переделан на веб.

## Как устроено приложение

### Экраны

| Режим | Описание |
|-------|----------|
| **В холодосе** (`status = active`) | продукты, которые есть дома |
| **Список покупок** (`status = finished`) | продукты, которые нужно купить |
| **Рецепты** (🍳) | свои рецепты; что можно приготовить из продуктов в каталоге |
| **Настройки** (⚙️) | магазины, продукты, категории |

Переключение «холодос ↔ список» — кнопка в шапке (🧊 / 📋). Рецепты — отдельная кнопка 🍳 рядом. На вкладке рецептов нет строки поиска продуктов и фильтра магазина.

### Действия с продуктом

| Жест | Где | Что делает |
|------|-----|------------|
| Поле «Найти или добавить…» | обе вкладки | поиск + добавление нового продукта |
| **+** (слева) | холодос | продукт закончился → в список покупок |
| **Чекбокс** | список покупок | куплено → обратно в холодос |
| Тап по продукту | настройки | modal с настройками (название, категория, мастхэв, магазины, удаление) |

- Названия сохраняются в **нижнем регистре**.
- Дубликаты запрещены (без учёта регистра); при совпадении показывается подсказка, где продукт уже есть, с переходом на другую вкладку. То же при **переименовании** в настройках.
- При добавлении на списке покупок с выбранным магазином — подсказка «Добавлено в список покупок» (без автоматического исключения магазина).

### Поиск и фильтры

- Общий `searchQuery` в `App.tsx` — строка поиска сохраняется при переключении вкладок.
- Если на текущей вкладке ничего не найдено, но есть совпадение на другой — показывается блок «По запросу … в списке покупок / в холодосе» (только по основному списку на вкладке покупок).
- Фильтр **«Мастхэв»** — в панели фильтров (воронка); показывает только избранные продукты.
- Фильтр по категориям — кнопка-иконка (воронка); при фильтре «Все» включены **липкие заголовки категорий** при скролле.

Карточки **компактные**: одна строка, название без категории на карточке (категория — в заголовках).

### Магазины и blacklist

Логика **blacklist**: продукт по умолчанию «можно купить везде»; в настройках отмечаются только магазины, где **не** планируешь покупать.

| Что | Где |
|-----|-----|
| Справочник магазинов (CRUD, порядок ↑↓) | Настройки → Магазины |
| «Не покупаю в: ☐ …» | при добавлении продукта («Ещё») и в форме настройки продукта |
| Фильтр «Сейчас: [магазин ▾]» | только список покупок |
| Блок «Купить в другом месте (N)» | свёрнутый список excluded-продуктов при выбранном магазине |

- Если магазинов нет — переключатель на списке покупок скрыт, приложение работает как раньше.
- Счётчик: `8 из 11 для Пятёрочки` — сколько позиций доступно в магазине из общего числа в списке.
- Последний выбранный магазин в фильтре — **localStorage** (только на этом устройстве).

### Мастхэв

Избранные продукты (`is_favorite` в Supabase). Закрепление — ★ в **форме настройки продукта** (Настройки → Продукты → тап). На вкладках — фильтр «Мастхэв» и бейдж на карточке. Лимит — **8** продуктов.

### Категории

- При добавлении — автоопределение по ключевым словам (`src/lib/detectCategory.ts`).
- Конфиг категорий (кастомные, порядок, переименование) — **localStorage** (`src/lib/categoryConfig.ts`), не Supabase.
- В настройках: добавление / переименование / удаление категорий (удалённая → `прочее` у продуктов), порядок ↑↓.

### Настройки

Главный экран — три плашки со счётчиками: **Продукты**, **Магазины**, **Категории** (в этом порядке). Тап по плашке открывает соответствующий список; «← Настройки» возвращает к плашкам, «← Назад» — в приложение.

| Раздел | Что внутри |
|--------|------------|
| **Продукты** | поиск по названию; тап по продукту — **modal** поверх списка (скролл сохраняется) |
| **Магазины** | CRUD, порядок ↑↓ |
| **Категории** | добавление / переименование / удаление (удалённая → `прочее` у продуктов), порядок ↑↓ |

**Modal настройки продукта** (`ProductSettingsForm`):

- закрытие — **×** в правом верхнем углу или **тап по затемнённой области** вокруг карточки;
- при несохранённых изменениях — confirm «Изменения не сохранены…»;
- **название** — тап по заголовку для редактирования, сохранение вместе с остальным по кнопке «Сохранить»;
- **удаление** — ссылка «Удалить продукт» внизу карточки, затем подтверждение (исчезнет из холодоса и списка покупок);
- категория, «часто покупаю» (★), «Не покупаю в: ☐ …»;
- после успешного сохранения строка продукта **кратко подсвечивается** в списке (~2 с).

### Рецепты

Только **свои** рецепты, вручную. Ингредиенты — только из каталога продуктов app (`product_id`); без граммовок, порций и внешних ссылок.

| Действие | Где | Что делает |
|----------|-----|------------|
| **+ Рецепт** | вкладка рецептов | форма создания |
| Тап по рецепту | список | детальный просмотр (modal) |
| **Редактировать** | просмотр рецепта | форма редактирования |
| Поиск | вкладка рецептов | фильтр по названию (локально) |

**Секции списка** (пересчитываются при смене продуктов в холодосе / покупках):

Сначала блоки **Завтраки** и **Обеды** (`meal_type`: `breakfast` | `lunch`). До **12:00** сверху «Завтраки», ниже «Обеды» (второй блок можно свернуть); с **12:00** — наоборот. Оба типа всегда видны. Порядок — `src/lib/recipeMealTime.ts` (локальное время устройства).

Внутри каждого блока:

| Секция | Условие |
|--------|---------|
| **Можно сейчас** | все ингредиенты `active` |
| **Не хватает** | хотя бы один ингредиент `finished`; подпись «нет: …» и «в списке покупок»; сортировка по числу недостающих |
| **Нужно поправить** | ингредиент удалён из app (`product_id = null` после удаления продукта) |

**Форма рецепта** (`RecipeForm`, modal как у продукта):

- название, **тип** (завтрак / обед), инструкция, время (минуты; `0` = не указано);
- при создании тип по умолчанию — актуальный по времени суток;
- ингредиенты — multiselect из каталога (🧊 / 📋 в picker);
- выбранные ингредиенты — чипы под поиском с **×** (можно убрать, в т.ч. «удалённый ингредиент»);
- минимум 1 ингредиент; удаление рецепта — с подтверждением.

**Просмотр рецепта** (`RecipeDetail`): тип приёма пищи, ингредиенты с ✅ / ❌ / ⚠️; **недостающие (❌) — вверху списка**; кнопка «Редактировать».

Удаление продукта в настройках **не удаляет** рецепт: ингредиент «ломается» (`ON DELETE SET NULL`), рецепт уходит в «Нужно поправить». Кнопки «добавить в покупки» нет — finished уже в списке покупок.

## Где что хранится

| Данные | Где | Синхронизация между устройствами |
|--------|-----|----------------------------------|
| Продукты (имя, статус, категория, мастхэв, даты) | Supabase, `products` | да (один Supabase-проект) |
| Рецепты и ингредиенты | Supabase, `recipes`, `recipe_ingredients` | да |
| Справочник магазинов | Supabase, `stores` | да |
| Blacklist «не покупаю в» | Supabase, `product_store_exclusions` | да |
| Категории (кастом, порядок) | localStorage | нет |
| Выбранный магазин в фильтре списка покупок | localStorage | нет |
| Поисковая строка | state в `App.tsx` | нет (сессия браузера) |

Без авторизации: RLS в Supabase открыт для read/insert/update/delete (личный проект). Real-time нет — изменения с другого устройства видны после перезагрузки / переключения вкладки / выхода из настроек.

### Архитектурные особенности

- Два экземпляра `useProducts('active')` и `useProducts('finished')` и `useRecipes()` в `App.tsx`.
- После удаления продукта обновляются обе вкладки продуктов и рецепты (при выходе из настроек / на вкладке 🍳).
- При входе на вкладку рецептов перезагружаются продукты и рецепты (актуальные ready / missing).
- После «кончилось» / «куплено» рецепты пересчитываются через refresh списков.
- После выхода из настроек списки, exclusions и рецепты перезагружаются из Supabase.
- Удаление проверяет, что строка реально удалилась (`.delete().select('id')`).
- Удаление магазина или продукта каскадно удаляет связанные exclusions.

## База данных (Supabase)

### `products`

| Колонка | Тип | Описание |
|---------|-----|----------|
| `id` | uuid | первичный ключ |
| `name` | text | название (нижний регистр) |
| `status` | text | `active` или `finished` |
| `category` | text | по умолчанию `прочее` |
| `is_favorite` | boolean | мастхэв, по умолчанию `false` |
| `created_at` | timestamptz | когда добавили |
| `finished_at` | timestamptz | когда перенесли в список покупок |

### `stores`

| Колонка | Тип | Описание |
|---------|-----|----------|
| `id` | uuid | первичный ключ |
| `name` | text | название магазина |
| `sort_order` | integer | порядок в настройках и фильтре |
| `created_at` | timestamptz | когда добавили |

### `product_store_exclusions`

| Колонка | Тип | Описание |
|---------|-----|----------|
| `product_id` | uuid | FK → `products` (cascade delete) |
| `store_id` | uuid | FK → `stores` (cascade delete) |

Наличие строки = не покупать этот продукт в этом магазине. Первичный ключ — пара `(product_id, store_id)`.

### `recipes`

| Колонка | Тип | Описание |
|---------|-----|----------|
| `id` | uuid | первичный ключ |
| `title` | text | название |
| `meal_type` | text | `breakfast` или `lunch` |
| `instructions` | text | инструкция, по умолчанию `''` |
| `cook_time_minutes` | integer | время в минутах; `0` = не указано |
| `sort_order` | integer | порядок |
| `created_at` | timestamptz | когда добавили |

### `recipe_ingredients`

| Колонка | Тип | Описание |
|---------|-----|----------|
| `id` | uuid | первичный ключ |
| `recipe_id` | uuid | FK → `recipes` (cascade delete) |
| `product_id` | uuid, nullable | FK → `products` (**ON DELETE SET NULL**) |

Уникальность пары `(recipe_id, product_id)` — только для ненулевых `product_id`. Один продукт — один раз в рецепте.

### Миграция

| Ситуация | Файл |
|----------|------|
| Новая БД | `supabase/schema.sql` целиком |
| `products` уже есть, без магазинов | `supabase/migrate-stores.sql` |
| нужны рецепты | `supabase/migrate-recipes.sql` |
| ошибка `product_id violates not-null` при удалении продукта | `supabase/migrate-recipes-fix-null-product-id.sql` |
| рецепты без `meal_type` | `supabase/migrate-recipes-meal-type.sql` |

Не запускайте весь `schema.sql` повторно на существующей БД — policies для `products` уже созданы и вызовут ошибку.

## Запуск локально

### 1. Переменные окружения

```bash
cp .env.example .env
```

```
VITE_SUPABASE_URL=https://ваш-проект.supabase.co
VITE_SUPABASE_ANON_KEY=ваш-anon-key
```

Ключи: Supabase → **Project Settings → API** (нужен **anon public**, не `service_role`).

### 2. Dev-сервер

```bash
npm install
npm run dev
```

Открыть `http://localhost:5173`

### 3. С телефона в той же Wi‑Fi

```bash
npm run dev -- --host
```

### 4. Сборка

```bash
npm run build
npm run preview
```

## Деплой (Vercel)

1. Подключить репозиторий к Vercel.
2. В **Environment Variables** задать `VITE_SUPABASE_URL` и `VITE_SUPABASE_ANON_KEY`.
3. Сборка: `npm run build`, выход — `dist/` (см. `vercel.json`).

После смены `.env` локально перезапустить dev-сервер; на Vercel — redeploy.

## Частые проблемы

| Симптом | Что проверить |
|---------|---------------|
| `ERR_CONNECTION_REFUSED` на :5173 | не запущен `npm run dev` |
| «Не удалось загрузить данные» / invalid API key | `.env`, ключи Supabase, перезапуск dev |
| `policy "Anyone can read products" already exists` | выполнить `migrate-stores.sql`, а не весь `schema.sql` |
| Продукт в базе есть, в списке нет | нужная вкладка; фильтр **«Все магазины»**; `status` в БД |
| Продукт в «другом месте», а не в основном списке | выбран магазин + exclusion для него в настройках продукта |
| «Не удалось удалить продукт» | delete-политика в `schema.sql` |
| `product_id violates not-null` при удалении продукта из рецепта | выполнить `migrate-recipes-fix-null-product-id.sql` |
| Рецепт не обновился после удаления продукта | «Назад» из настроек или переключить вкладку 🍳 |
| Мастхэв / exclusions не обновились | «Назад» в настройках — списки перезагрузятся |
| На другом устройстве нет магазинов | тот же Supabase-проект в `.env` / Vercel env |

## История и планы

Краткая эволюция: Expo → веб (Vite); категории и CRUD → UX (поиск, компактные карточки, липкие категории, мастхэв) → **магазины, blacklist, фильтр списка покупок** → **рецепты** (ready / missing / broken, завтрак/обед по времени суток, CRUD).

Подробное ТЗ по рецептам — **[RECIPES_IMPLEMENTATION.md](./RECIPES_IMPLEMENTATION.md)**.

Будущие фичи — в **[ROADMAP.md](./ROADMAP.md)** (карточка продукта из списка, preferred store, маршрут на день, bulk-exclusions и др.).
