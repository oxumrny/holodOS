# holodOS

Личный веб-трекер продуктов в холодильнике: что есть дома и что нужно купить. Проект для себя и близких; репозиторий: [github.com/oxumrny/holodOS](https://github.com/oxumrny/holodOS).

---

## Стек

| Слой | Технология |
|------|------------|
| Фронтенд | Vite 6 + React 19 + TypeScript |
| Бэкенд / БД | Supabase (PostgreSQL + REST API) |
| Стили | обычный CSS (файлы рядом с компонентами) |
| Деплой | Vercel (`vercel.json`, сборка в `dist/`) |

Раньше проект был **Expo + React Native**; переделан на веб.

## Как устроено приложение

### Экраны

| Режим | Описание |
|-------|----------|
| **В холодосе** (`status = active`) | продукты, которые есть дома |
| **Список покупок** (`status = finished`) | продукты, которые закончились |
| **Настройки** (⚙️) | категории и закрепление в «Часто» |

Переключение «холодос ↔ список» — одна кнопка в шапке (🧊 / 📋), не отдельные вкладки.

### Действия с продуктом

| Жест | Где | Что делает |
|------|-----|------------|
| Поле «Найти или добавить…» | холодос | поиск + добавление нового продукта |
| Поиск | список покупок | только фильтрация |
| **+** (слева) | холодос | продукт закончился → в список покупок |
| **Чекбокс** | список покупок | куплено → обратно в холодос |
| **×** (справа) | обе вкладки | удалить навсегда из Supabase |
| **★** | настройки | закрепить / убрать из «Часто» |

- Названия сохраняются в **нижнем регистре**.
- Дубликаты запрещены (без учёта регистра); при совпадении показывается подсказка, где продукт уже есть, с переходом на другую вкладку.

### Поиск и фильтры

- Общий `searchQuery` в `App.tsx` — строка поиска сохраняется при переключении вкладок.
- Если на текущей вкладке ничего не найдено, но есть совпадение на другой — показывается блок «По запросу … в списке покупок / в холодосе».
- Фильтр по категориям — кнопка-иконка (воронка); при фильтре «Все» включены **липкие заголовки категорий** при скролле.
- Карточки **компактные**: одна строка, название без категории на карточке (категория — в заголовках).

### Блок «Часто»

Короткий список вверху обеих вкладок. Данные **не в Supabase**, а в **localStorage** (`src/lib/frequentProducts.ts`):

| Ключ | Содержимое |
|------|------------|
| `holodos-pinned-products` | вручную закреплённые (★ в настройках) |
| `holodos-product-activity` | счётчик действий (+ / «куплено») |

Правила:

1. **Ручное закрепление** — только в **Настройках → Продукты** (звёздочка ★). На вкладках списков звёздочка не кликается.
2. **Авто-добавление** — продукт попадает в «Часто» после **2+ действий** (закончился / куплено), если не закреплён вручную.
3. **Лимит отображения** — не больше **8** продуктов в блоке «Часто» (сначала закреплённые, потом авто). Отдельно при закреплении в настройках тоже действует лимит 8 — это UX-ограничение, не требование БД.
4. На карточках в блоке «Часто» — **readonly** бейдж **musthave** (параллелограмм, золотой градиент).
5. При удалении продукта из базы сбрасываются и pin, и activity (`removeProductTracking` в `App.tsx`).

### Категории

- При добавлении — автоопределение по ключевым словам (`src/lib/detectCategory.ts`).
- Конфиг категорий (кастомные, порядок, переименование) — **localStorage** (`src/lib/categoryConfig.ts`), не Supabase.
- В настройках: смена категории у продукта, добавление / переименование / удаление категорий (удалённая → `прочее` у продуктов).

## Где что хранится

| Данные | Где |
|--------|-----|
| Продукты (имя, статус, категория, даты) | Supabase, таблица `products` |
| Категории (кастом, порядок) | localStorage |
| «Часто» (pin + activity) | localStorage |
| Поисковая строка | state в `App.tsx` (сессия браузера) |

Без авторизации: RLS в Supabase открыт для read/insert/update/delete (личный проект).

### Архитектурные особенности

- Два экземпляра `useProducts('active')` и `useProducts('finished')` в `App.tsx`.
- После удаления обновляются обе вкладки; из localStorage убирается tracking продукта.
- `musthaveRevision` в `App` — пересчёт блока «Часто» после выхода из настроек.
- `frequentRevision` в `ProductList` — пересчёт после действий (+ / куплено), когда растёт activity.
- Удаление проверяет, что строка реально удалилась (`.delete().select('id')`).

## База данных (Supabase)

Таблица `products`:

| Колонка | Тип | Описание |
|---------|-----|----------|
| `id` | uuid | первичный ключ |
| `name` | text | название (нижний регистр) |
| `status` | text | `active` или `finished` |
| `category` | text | по умолчанию `прочее` |
| `created_at` | timestamptz | когда добавили |
| `finished_at` | timestamptz | когда перенесли в список покупок |

Скрипт: `supabase/schema.sql`.

## Запуск локально

### 1. Переменные окружения

```bash
cp .env.example .env
```

```
VITE_SUPABASE_URL=https://ваш-проект.supabase.co
VITE_SUPABASE_ANON_KEY=ваш-anon-key
```

Ключи: Supabase → **Project Settings → API** (нужен **anon public**, не `service_role`).

### 2. Dev-сервер

```bash
npm install
npm run dev
```

Открыть `http://localhost:5173`

### 3. С телефона в той же Wi‑Fi

```bash
npm run dev -- --host
```

### 4. Сборка

```bash
npm run build
npm run preview
```

## Деплой (Vercel)

1. Подключить репозиторий к Vercel.
2. В **Environment Variables** задать `VITE_SUPABASE_URL` и `VITE_SUPABASE_ANON_KEY`.
3. Сборка: `npm run build`, выход — `dist/` (см. `vercel.json`).

После смены `.env` локально перезапустить dev-сервер; на Vercel — redeploy.

## Частые проблемы

| Симптом | Что проверить |
|---------|---------------|
| `ERR_CONNECTION_REFUSED` на :5173 | не запущен `npm run dev` |
| «Не удалось загрузить данные» / invalid API key | `.env`, ключи Supabase, перезапуск dev |
| Продукт в базе есть, в списке нет | нужная вкладка; фильтр **«Все»**; `status` в БД |
| «Не удалось удалить продукт» | delete-политика в `schema.sql` |
| «Часто» пустое после закрепления | выйти из настроек («Назад») — обновится `musthaveRevision` |
| Закрепления пропали на другом устройстве | pin/activity только в localStorage этого браузера |
